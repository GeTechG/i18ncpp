<objective>
## Goal
Close AEGIS findings M1 (configure() partial mutation) and M2 (string_view UB / Windows Unicode path) by making `configure()` transactional (copy-and-swap with commit-window cache invalidation), widening the `loadLocale`/`mergeLocale`/`load(json)` catch clauses to `json::exception`, and routing locale-file `ifstream` construction through `std::filesystem::path`.

## Purpose
The library's public exception contract (`include/i18ncpp.h:45-48`) promises `I18NError` is the only thrown type, but `json::type_error` escapes `configure()` today — and when it does, `defaultConfig` is half-mutated and the caches are not cleared, producing silent wrong formatting until the next `reset()`. Separately, `loadLocale`/`mergeLocale` pass `string_view::data()` to an `ifstream(const char*)` overload, which is both a latent UB footgun (non-null-terminated views) and a silent Windows Unicode-path failure. Both defects are boundary-correctness issues: the library's trust boundary with nlohmann/json, and the library's trust boundary with the filesystem.

## Output
- `src/i18ncpp.cpp`: `configure()` rewritten to stage into a local `FormatConfig tmp`, catch `json::exception`, commit via move + cache-clear; catches in `loadLocale`/`mergeLocale` widened; `load(const json&)` body wrapped; `loadLocale`/`mergeLocale` use `std::filesystem::path(filePath)` for `ifstream` construction.
- `tests/fixtures/formats_wrong_type.json`: hostile fixture (shared with P-005 in Phase 3).
- `tests/test_locale_mgmt.cpp`: three new GTest cases covering the exception contract, cache invariant post-throw, and non-null-terminated `string_view` path acceptance.
- `tests/CMakeLists.txt`: copy the new fixture to the test binary directory (if not already globbed).
</objective>

<acceptance_criteria>

## AC-1: configure() provides strong exception safety
```gherkin
Given a fully-initialized I18N instance with a populated formatCache_ and translationCache_
When configure() is called with a JSON object where "_formats.currency.symbol" is an array (wrong type)
Then configure() throws I18NError (not json::type_error)
And defaultConfig equals its pre-call state byte-for-byte (rollback)
And formatCacheSize() equals the pre-call size (commit window never reached, so cache remains valid against unchanged source)
```

## AC-2: loadLocale / mergeLocale / load(json) surface only I18NError
```gherkin
Given an I18N instance
When loadLocale, mergeLocale, or load(json) encounters a json::type_error (not just parse_error) during its body
Then the call throws I18NError with a descriptive message
And no nlohmann::json exception type escapes to the caller
```

## AC-3: loadLocale accepts non-null-terminated string_view and Unicode paths
```gherkin
Given a std::string_view constructed via substr() over a backing buffer with trailing garbage
When loadLocale("en", sv) is called with a valid locale fixture path in the view
Then the file is opened correctly (no read past the view's end)
And the fixture loads successfully
And on Windows, a path containing non-ASCII characters opens without silent failure
```
