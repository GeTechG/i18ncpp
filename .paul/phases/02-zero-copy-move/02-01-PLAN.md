<objective>
## Goal
Eliminate unnecessary string allocations in key lookup paths and data loading by introducing transparent hashing for heterogeneous map lookup and move semantics for JSON string extraction.

## Purpose
Benchmarks (Phase 1) identified key lookup and interpolation as hot paths. Every `getTranslationData` call currently converts `string_view` → `string` just to query the map. Every `flattenJson` call copies strings from parsed JSON even when the JSON is a temporary. These are the two lowest-hanging zero-copy wins.

## Output
- Modified `include/i18ncpp.h` — transparent hash/equal types, updated map typedefs
- Modified `src/i18ncpp.cpp` — zero-copy lookups, move-based JSON extraction, reduced temporaries
</objective>

<task type="auto">
  <name>Task 1: Add transparent hash/equal and convert maps to heterogeneous lookup</name>
  <files>include/i18ncpp.h, src/i18ncpp.cpp</files>
  <action>
    1. In i18ncpp.h, add before the I18N class definition:
       - `struct StringHash` with `using is_transparent = void;` that hashes both `std::string` and `std::string_view` via `std::hash<std::string_view>`
       - `struct StringEqual` with `using is_transparent = void;` that compares any combination of string/string_view

    2. Change map types in I18N class:
       - `localesData`: `std::unordered_map<std::string, std::unordered_map<std::string, std::string, StringHash, StringEqual>, StringHash, StringEqual>`
       - `formatConfigs`: `std::unordered_map<std::string, FormatConfig, StringHash, StringEqual>`
       - `formatCache_`: `std::unordered_map<std::string, std::string, StringHash, StringEqual>`

    3. In src/i18ncpp.cpp, update `getTranslationData`:
       - Remove `std::string localeStr(locale);` and `std::string keyStr(key);` temporaries
       - Use `localesData.find(locale)` and `localeIt->second.find(key)` directly (transparent lookup)

    4. Update all other map lookups that convert string_view to string just for `.find()`:
       - `setLocale`: `formatConfigs.find(locale)` instead of `formatConfigs.find(localeStr)`
       - `formatCache_.find(cacheKey)` calls are already using std::string keys so no change needed there

    Avoid: Changing the public API signatures. Avoid `std::hash<std::string>` — use `std::hash<std::string_view>` for all hashing since `std::string` is implicitly convertible to `std::string_view`.
  </action>
  <verify>Build with CMake and run all tests: `cmake --build build --config Release && ctest --test-dir build -C Release`</verify>
  <done>AC-1 satisfied: getTranslationData performs map lookups with string_view directly, no string allocation</done>
</task>

<task type="auto">
  <name>Task 2: Move semantics for JSON string extraction in flattenJson</name>
  <files>include/i18ncpp.h, src/i18ncpp.cpp</files>
  <action>
    1. Add a non-const overload of `flattenJson` that takes `json&` (mutable reference):
       - Signature: `void flattenJson(const std::string& prefix, json& node, std::unordered_map<std::string, std::string, StringHash, StringEqual>& flatMap);`
       - Use `sub.value().get_ref<std::string&>()` with `std::move()` to extract string values
       - For the leaf object branch, move each sub-value string
       - For plain string values, move the string out

    2. Keep the existing const overload for `load(const json&)` which receives externally-owned data.

    3. Update callers to use the mutable overload where the JSON is locally owned:
       - `loadLocale`: passes locally parsed `json data` → use mutable overload
       - `mergeLocale`: passes locally parsed `json data` → use mutable overload
       - `load`: passes `const json&` from user → keep const overload

    Avoid: Breaking `load(const json&)` — the caller still owns that data. Only move from JSON we parsed ourselves.
  </action>
  <verify>Build and run all tests. Verify loadLocale/mergeLocale still work correctly with the move path.</verify>
  <done>AC-2 satisfied: loadLocale and mergeLocale move strings from parsed JSON instead of copying</done>
</task>

<task type="auto">
  <name>Task 3: Reduce temporary string allocations in lookup hot paths</name>
  <files>src/i18ncpp.cpp</files>
  <action>
    1. In `tr(string_view key, const vector<string>& params)`:
       - Avoid constructing `std::string keyStr(key)` upfront
       - For composite key lookups like `keyStr + ".other"`, build the composite key in a reusable buffer string declared once, or use the transparent lookup with a constructed string only when needed

    2. In `trPlural(string_view key, int count, const vector<string>& params)`:
       - Same pattern: avoid eager `std::string keyStr(key)` construction
       - Reuse a single buffer string for building composite keys (key + "." + pluralForm, key + ".other", etc.)
       - Use `.clear()` + `.append()` on the buffer instead of constructing new strings

    3. In `translate(string_view key, const json& params)`:
       - Same buffer pattern for composite key construction

    4. In `getFallbacks`:
       - Change `std::unordered_map<std::string, bool> seen` to `std::unordered_set<std::string, StringHash, StringEqual>` to avoid storing bool values and enable transparent lookup

    Avoid: Over-engineering — don't change function signatures or the public API. Focus on reducing allocations within existing function bodies.
  </action>
  <verify>Build and run all tests. All 82 tests must pass.</verify>
  <done>AC-3 satisfied: all tests pass, composite key construction uses fewer allocations</done>
