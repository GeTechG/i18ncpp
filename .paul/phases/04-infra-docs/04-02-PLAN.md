<objective>
## Goal
Install a GitHub Actions CI workflow that builds and tests i18ncpp on a pinned OS/compiler matrix and surfaces status via a README badge, so every push and PR to master is verified against the same contract documented in CLAUDE.md.

## Purpose
v0.4.1 Hardening & Infrastructure's last missing piece is automated verification. Phases 1-3 produced hardening fixes and new test suites (boundary correctness, cache invariant, hostile input, TSan-gated thread test) that currently only run on the author's Windows box. CI locks that contract in place before the next feature cycle — regressions on any matrix cell must fail the build, not leak to master.

## Output
- `.github/workflows/ci.yml` — matrix build + test workflow
- README.md — CI status badge at the top
</objective>

<task type="checkpoint:decision" gate="blocking">
  <decision>CI matrix dimensions for the first workflow</decision>
  <context>
    First CI run needs to cover the project's hard constraints (C++20, MSVC multi-config, Conan-driven build, BUILD_TESTS=ON) without becoming a maintenance sink. Four dimensions to pin: OS set, build config, benchmarks, and TSan.
  </context>
  <options>
    <option id="minimal">
      <name>Minimal: Ubuntu + Windows, Release only, no benchmarks, no TSan</name>
      <pros>
        - Two cells, fast CI (&lt;10min each)
        - Covers GCC/Clang on Ubuntu and MSVC multi-config on Windows — the two compilers the code actually needs to work on
        - Matches what the author already runs locally
        - Zero new infra
      </pros>
      <cons>
        - No macOS coverage (AppleClang untested)
        - Debug config not exercised — asserts and iterator debug checks never run in CI
        - TSan test stays local-only, so thread-safety regressions won't be caught by CI until someone remembers to rerun locally
      </cons>
    </option>
    <option id="standard">
      <name>Standard: Ubuntu + Windows + macOS, Release only, no benchmarks, no TSan</name>
      <pros>
        - Three-platform coverage catches AppleClang-specific issues (libc++ vs libstdc++, different std::format maturity)
        - Still single config per cell — fast
      </pros>
      <cons>
        - macOS runner minutes cost more (public repo is free, but queues are slower)
        - No proof the code hasn't shipped macOS-specific bugs yet — adding the cell will likely surface issues that delay this plan
      </cons>
    </option>
    <option id="thorough">
      <name>Thorough: Ubuntu + Windows, Release + Debug, no benchmarks, TSan job on Ubuntu</name>
      <pros>
        - Debug config catches assertion failures and iterator debug issues
        - TSan-gated job runs the thread-safety regression test automatically (unblocks Phase 2's TSan infra, which is currently opt-in local-only)
        - Still two OSes — no macOS surprise costs
      </pros>
      <cons>
        - Four+ cells, longer CI runtime
        - TSan job needs special CMake flags plumbing — non-trivial to get right on first attempt
        - Likely reveals new issues that push this plan past its scope
      </cons>
    </option>
  </options>
  <resume-signal>Select: minimal, standard, or thorough. Chosen option will be recorded in Task 2's action block before YAML is written.</resume-signal>
</task>

<task type="auto">
  <name>Task 2: Create GitHub Actions workflow</name>
  <files>.github/workflows/ci.yml</files>
  <action>
    Create a new workflow file at `.github/workflows/ci.yml` using the matrix chosen in Task 1.

    Required workflow shape (regardless of matrix choice):
    - `name: CI`
    - Triggers: `push` to master, `pull_request` to master, plus `workflow_dispatch`
    - One `build` job with `strategy.matrix` populated from the Task 1 decision
    - Python setup (actions/setup-python@v5) for Conan
    - `pip install conan` (pin to major version, e.g. `conan>=2.0,<3`)
    - `conan profile detect --force` on first run
    - `conan install . --build=missing -s build_type=Release` (adjust `build_type` if Task 1 added Debug)
    - `cmake --preset conan-default -DBUILD_TESTS=ON`
    - `cmake --build --preset conan-release` (or matching preset for Debug)
    - `ctest --test-dir build -C Release --output-on-failure` (match `-C` to config)
    - `fail-fast: false` on the matrix so one cell's failure doesn't cancel the others
    - On Windows: do NOT override the generator; let Conan/CMake pick MSBuild so multi-config works
    - On Ubuntu: install build-essential + cmake + ninja via apt-get before Conan

    Avoid:
    - Do NOT add caching in this plan (ccache/sccache/actions/cache) — defer to a follow-up
    - Do NOT run benchmarks in CI (scope limit)
    - Do NOT add coverage/sonar/coverity uploads (out of scope)
    - Do NOT trigger on tags or schedule — push + pull_request + workflow_dispatch only
    - Do NOT skip hooks, secrets, or signing — none are used
    - Do NOT use third-party composite actions for Conan; call Conan directly (smaller supply-chain surface)

    Record the resolved matrix choice from Task 1 as a top-of-file comment: `# Matrix: {choice} — see .paul/phases/04-infra-docs/04-02-PLAN.md AC-4`
  </action>

    4. Grep check: no references to benchmarks, coverage, or sonarqube
  </verify>
  <done>AC-1, AC-2, AC-4 satisfied: YAML exists, canonical Conan-driven build commands are used, and the matrix matches the Task 1 decision.</done>
