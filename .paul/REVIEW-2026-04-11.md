# Code Review 2026-04-11

## Confirmed Findings

1. `load()` leaves stale translations behind instead of replacing a locale.
   - File: `src/i18ncpp.cpp:131`
   - Problem: `flattenJson("", it.value(), localesData[localeStr])` writes into the existing map without clearing it first.
   - Impact: loading the same locale twice through `load(json)` preserves keys that were removed from the new payload.

2. Locale format configuration leaks across locales.
   - Files: `src/i18ncpp.cpp:139`, `src/i18ncpp.cpp:155`, `src/i18ncpp.cpp:168`, `src/i18ncpp.cpp:1131`
   - Problem: `configure()` builds from the current `defaultConfig`, and `setLocale()` only swaps `defaultConfig` when a locale-specific format config exists.
   - Impact:
     - a later locale can inherit formatting fields from an earlier locale;
     - switching to a locale with no `_formats` can keep the previous locale's formatting rules instead of resetting to defaults.

3. `trv()` and `trPluralv()` serialize `bool` as `"1"` / `"0"` instead of `"true"` / `"false"`.
   - File: `include/i18ncpp.h:260`
   - Problem: `std::is_arithmetic_v<T>` matches before the dedicated bool branch.
   - Impact: textual boolean interpolation is wrong for variadic helpers.

4. `keyExists()` does not match real lookup behavior.
   - Files: `src/i18ncpp.cpp:691`, `src/i18ncpp.cpp:743`, `src/i18ncpp.cpp:824`
   - Problem: it checks only the exact flat key, while `tr()` and `trPlural()` also resolve plural/variant suffixes and fallback paths.
   - Impact: `keyExists("items_plural")` can return `false` even though lookup succeeds.

5. The implementation is not portable across common C++20 toolchains.
   - Files: `src/i18ncpp.cpp:969`, `src/i18ncpp.cpp:1091`
   - Problem: it uses MSVC-specific `localtime_s` calling style and depends on `std::format`.
   - Impact: GCC/Clang builds are likely to fail or need compatibility work, while the top-level CMake does not document that limitation.
